# 多 AI 讨论室 · 演进方向：从"聊天室"到"AI 议会（MCP Council）"

> 本文记录项目的方向性转折：把消费者从"人"换成"另一个 AI"。
> 这是回答"这个项目意义何在"的文档，不是施工清单（施工见 [ROADMAP.md](./ROADMAP.md)）。

---

## 一、动机：当前形态的诚实评估

现有形态是"人在网页上看多个 AI 讨论一个话题"。它是个不错的 demo，但作为产品**意义有限**：

1. **新鲜感驱动，没有反复出现的真实需求。** "看 3 个 AI 聊天"看一两次就够了，没人每天用。
2. **异构卖点尚未兑现。** 目前三个 agent 常接同一个模型，本质是"一个模型分饰多角"，观点差异全靠人设硬撑，不是真正不同厂商大脑的碰撞。

结论：**当前形态不该是终点。** 它的编排引擎（orchestrator）是有价值的资产，但"给人看的聊天"不是它的归宿。

---

## 二、转折：消费者从"人"换成"AI"

核心 idea：**让 AI agent（Claude Code、Codex 等 CLI）原生调用这个项目。**

场景：Claude Code 干活时遇到一个**拿不准的判断**——"方案选 A 还是 B"、"这段设计有没有坑"、"这个架构决策靠不靠谱"——它把**上下文 + 要判断的问题**发给本项目，项目召集 DeepSeek、通义、GLM 等**不同厂商的模型**讨论/辩论，再把**结构化结论**返回给 Claude，供其继续推理或行动。

一句话定位：

> **「硬决策时，调用方 AI 可以请求的『多模型第二意见 / 专家会诊』服务。」**

为什么这次"有用"了：
- 产出从"给人看的娱乐"变成"**喂给另一个 agent 推理的输入**"——这是有真实、反复需求的。
- **终于用上异构这个核心卖点**：Claude 自己想不到的盲区，由其它厂商模型来补。多智能体辩论（multi-agent debate）在研究上被验证能提升答案质量。
- 形态现成：做成一个 **MCP server**，暴露一个工具（如 `convene_council`）。编排引擎已就绪，只需在现有网页 HTTP/SSE 入口之外，**再开一个给 AI 调的 MCP 入口**。

---

## 三、关键认知：它是什么，不是什么

### 它不是 Codex 的替代品

| | Codex（被 Claude 调用） | 本项目 council |
|---|---|---|
| 本质 | **完整 agent**，有手有脚 | 一组 **chat 模型**，泡在罐子里的脑子 |
| 能力 | 自己读文件、跑命令、翻全仓库 | 只对**喂进来的文字**推理 |
| 适合 | **调查类**："这 bug 你去查查" | **判断类**："这个决策选哪个" |

**所以"council 看不到全部上下文"不是缺陷，是它本来就不该是那个角色。** 给 council 也加文件系统权限 = 做一个又慢又贵还跑五份的 Codex 复制品，没意义。

### "靠调用者 curate 上下文"是设计，不是退而求其次

- 判断一个决策**通常不需要整个仓库**，需要的是把决策点描述清楚。如同**医生会诊看病历，不重新检查病人**。
- 逼调用方把"现状 + 问题"提炼成一段话，**这个提炼动作本身就是高质量咨询的一半**；塞太多原始上下文反而稀释判断。
- 速度、成本、隐私三条也都要求"按需喂料"，这才是能扇出到多模型的唯一可行方式。

## 四、核心风险与对策

### 风险 1：慢和贵
一场讨论 = 十余次串行 LLM 调用，分钟级延迟、十倍 token。调用方要快答案，不会等。
- **对策：快模式**（fast）——单轮、并行问、无主持人；与现有慢模式（多轮、主持人调度）并存。

### 风险 2：多模型 ≠ 更好答案，可能只是更多废话
真正的价值是**暴露分歧（divergence 才是信号）**，不是磨出一个圆滑共识。
- **对策：结构化输出**——返回"哪里有分歧 / 各方最强论据 / 带条件的建议 / 置信度"，而非一段顺滑总结。

### 风险 3（最深）：盲区问题 —— GIGO
调用方 AI 的盲区，恰恰决定了它会**漏掉什么上下文**；而它找 council，本就是想补盲区。结果它按自己的理解喂料，把真正出问题的点漏掉，council 拿残缺信息自信地给错答案。
- **对策（让整个设计成立的一招）：允许 council 成员"拒答 / 反问"。**
  - 每个模型可以回："光这些判断不了，你没说 X" / "你这假设了 Y，成立吗？"
  - **多个模型共同指向同一个缺口** → 这本身就是返回给调用方的**最高价值信号**："你的问题框架有盲区，先补这块再来问"。一次性、缺上下文的限制，反而成了**探测盲区的探针**。
  - 优雅的进阶结构（v1 不做，方向对）：**council 是脑，调用方是眼睛和手**。council 说"我要看锁的代码"→ 调用方（有文件权限）去读了喂回来 → 迭代。council 不用长手，照样拿到点名要的上下文。

### 风险 4：异构是前提
全是同一个模型，council 就是回音壁。**必须真配上不同厂商的模型**，这套才有意义。

## 五、模拟样例（说明设计意图）

**调用**（Claude 故意漏掉关键判断点"并发量/冲突频率"）：
```
question: 库存扣减用乐观锁还是悲观锁?
context:  电商下单扣库存,多请求并发会超卖。我倾向乐观锁(version+CAS),觉得性能好。
mode:     fast
```

**三个异构模型的反应**（要点）：
- DeepSeek：低冲突乐观锁好；**秒杀高冲突会活锁**，该用悲观锁/Redis。缺冲突频率，无标准答案。
- 通义：换个层次——**库存前置 Redis 原子扣减**，DB 只落账。上下文不足，先补 QPS。
- GLM（红队）：**"觉得性能好"是直觉不是依据**。没 QPS、没重试上限，此问应被打回。

**返回给调用方的结论**（人话版）：
> ⚠️ 暂时无法下结论。原因：未说明并发量（日常下单 or 秒杀）——三个模型都指出缺这个。
> 若硬要建议：日常→乐观锁；秒杀→Redis 预扣减。
> 👉 先补充：大概多少 QPS？是否秒杀场景？

**这个样例证明的事**：盲区被多方共同探针抓到；council 合法"拒答+反问"而非硬答；异构带来真正不同的切入（补细节 / 换方案 / 做红队）。

## 六、双面孔输出：给 AI 一套，给人一套

同一场讨论结果要出**两个面孔**：
- **给 AI**：结构化 JSON（`verdict` / `divergence` / `missing_context` / `recommendation` / `confidence`），机器逐字段抓取、好行动。
- **给人**：大白话排版（现有网页 UI 已胜任），人一眼看懂。

两者来自同一次编排，仅渲染层不同。

## 七、分阶段计划（粗粒度，细化进 ROADMAP）

| 阶段 | 内容 | 价值 |
|---|---|---|
| **A. MCP 入口** | 把编排引擎包成 MCP server，暴露 `convene_council(question, context, mode)`；复用现有 orchestrator | Claude Code / Codex 能原生调用 |
| **B. 快模式** | 单轮 + 并行 + 无主持人，秒级返回 | 解决"慢和贵"，让 AI 愿意调 |
| **C. 结构化输出** | council 产出 `verdict / divergence / missing_context / recommendation / confidence` | 解决"更多废话"，让结果可被行动 |
| **D. 拒答/反问机制** | 成员可返回"信息不足 + 缺什么"；多方共识缺口高亮 | 堵住 GIGO 与盲区——**最关键** |
| **E. 异构默认阵容** | 预置一套真正多厂商的 council 模板，引导用户配齐 | 兑现异构卖点 |
| **F.（探索）脑-手迭代** | council 点名要上下文 → 调用方去取 → 回喂 | 突破"一次性喂料"的上限 |

> 网页"人来用"的形态**不废弃**，与 MCP"AI 来用"的形态共享同一编排引擎，双入口并存。

- 异构阵容的默认模板具体选哪几家。
