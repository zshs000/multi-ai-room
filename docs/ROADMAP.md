# 多 AI 讨论室 · 迭代路线图

> 本文件是项目的演进蓝图与施工总纲。每完成一个阶段，更新对应勾选与"迭代记录"。

## 一、产品愿景

一个让**不同厂商、不同模型的 AI 各自扮演角色，就同一话题同台讨论/辩论**的桌面级 Web 应用。
核心差异化：不是一个模型分饰多角，而是 DeepSeek、小米 MiMo、Claude、通义等**异构大脑真正同台**，观点天然有差异，讨论才有价值。

## 二、技术栈与架构决策

| 项 | 选型 | 理由 |
|---|---|---|
| 前端 | Vue 3 + Vite | 工程化、组件化，支撑产品长期演进 |
| 后端 | Node.js（原生 http，渐进引入必要依赖） | 轻量，LLM 流式生态好 |
| 通信 | REST + SSE（流式发言） | 简单、够用 |
| 多供应商 | **Provider / Agent 分离**（见下） | 凭证与使用者解耦，同厂商多人设不重复填 key |
| 多协议 | **适配器注册表**，按 `protocol` 字段路由 | 加新协议=新增一个适配器+注册一行，零侵入 |
| 持久化 | 阶段一仍用 JSON 文件，后期视情况升级 | 单机自用，先简单 |

### 核心数据模型（定调，细节见 API.md）

```
Provider { id, name, protocol: "openai"|"anthropic", baseUrl, apiKey, models: string[] }
Agent    { id, name, color, systemPrompt, providerId, model }
Config   { providers: Provider[], agents: Agent[], rounds, ... }
```

- Agent 不再内联 url/key，而是引用 `providerId` + 选一个 `model`。
- 适配器层吸收各协议差异（system 字段位置、请求结构、SSE 解析格式）。

## 三、四层迭代路线

### 第一层 · 地基重构（架构正确性）
目标：把原型升级为工程化骨架，多模型同台跑通。
- [x] 1.1 Vite + Vue 3 脚手架，前端迁移到组件化
- [x] 1.2 后端抽出适配器层：`OpenAIAdapter` + `AnthropicAdapter`，注册表按 protocol 路由
- [x] 1.3 数据模型升级为 Provider/Agent 分离，配置读写接口适配（含旧 config 自动迁移）
- [x] 1.4 讨论编排接 Provider：每个 Agent 用各自 provider+model 发言
- [x] 1.5 连通性测试接口 `POST /api/providers/test`（前置：地基调试期就需要它验证 key，不必靠真跑讨论）
- **验收**：
  - [x] 能配 ≥2 个不同协议的 provider（如 DeepSeek + Claude），3 个 agent 分属不同 provider，同台流式讨论跑通。
  - [x] 故意给某 provider 填错 key，该 agent 报错但整场不崩（验证 §6 错误隔离不变量）。

### 第二层 · Agent 与供应商可视化管理（产品灵魂）
目标：全程界面操作，不碰 config.json。
- [x] 2.1 Provider 管理：增删改查，预置模板（DeepSeek/通义/Kimi/智谱/小米/Claude 预填 baseUrl+protocol）
- [x] 2.2 Agent 管理：增删改人设/颜色，下拉选 provider（带"+新建供应商"内联入口），model 联动
- [x] 2.3 连通性测试的 UI 化：填完 provider 一键测试（复用 1.5 的接口），结果可视化展示
- [x] 2.4 阵容模板：辩论赛/头脑风暴/专家会诊等一键载入
- [x] 2.x 补强：Agent 发言顺序上移/下移
- **验收**：新用户零编辑文件，纯界面完成"配供应商→建角色→开聊"。✅

### 第三层 · 体验打磨（像样产品的质感）
目标：第一眼就觉得是成熟产品。
- [x] 3.1 Markdown 渲染（含代码高亮）——AI 输出不再是乱排的纯文本
- [x] 3.2 流式可中断：讨论中途"停止"按钮 + 中断标记
- [x] 3.3 加载指示、自动滚动优化（用户上滚不强制拽回）、错误可重试
- [x] 3.4 整体视觉升级：头像、深色模式、响应式
- **验收**：一轮完整讨论观感流畅，markdown 正确，能随时停。✅

### 第四层 · 讨论变"活" + 工具化（高级编排与留存）
目标：从"看 AI 讨论"升级为"参与并复用"。
- [x] 4.1 用户可插话：讨论中追问、@指定 Agent
- [x] 4.2 主持人模式：调度 AI 决定下一个谁发言、何时收尾（替代死板轮流）
- [x] 4.3 共识总结：末尾由总结 Agent 输出结论
- [x] 4.4 多会话持久化：历史会话列表、刷新不丢、可续聊
- [x] 4.5 导出讨论为 Markdown
- **验收**：可保存/重开多场讨论，支持人工插话与智能调度。✅

## 四、工作纪律

- 每完成一个编号小项即 `git commit`，提交信息说明"为什么"。
- 文档先行：接口变动先改 `API.md` 再写代码。
- 关键设计歧义先咨询子代理，重要文档/产出让子代理审查。
- 不入库密钥（config.json 已 gitignore）。
